# 🎯 Objective

Complete OverTheWire Bandit Level 13 while understanding every stage of SSH public key authentication rather than simply executing commands.

---

# Environment

Platform:
- Ubuntu 26.04 (WSL)
- OpenSSH 10.2
- OverTheWire Bandit

---

# Tasks Performed

## 1. Logged into Bandit13

Authenticated using the password obtained from the previous challenge.

---

## 2. Investigated the Provided SSH Private Key

Verified the existence of:

sshkey.private

Confirmed file type using Linux utilities.

Studied the structure of OpenSSH private keys.

---

## 3. Learned Identity File Authentication

Instead of default key lookup inside:

~/.ssh/

I explicitly instructed SSH to use:

ssh -i sshkey.private

Understanding the `-i` option was one of today's major learning milestones.

---

## 4. Authentication Debugging

When authentication initially appeared unsuccessful, I investigated the issue rather than searching for walkthroughs.

Used:

- ssh -v
- ssh -vv
- ssh -vvv

to analyse:

- Key loading
- Identity selection
- Host verification
- Authentication sequence

---

## 5. Linux Permission Investigation

Inspected file permissions.

Observed:

640

Instead of blindly changing permissions, I first investigated whether the issue actually originated from file permissions.

This reinforced an important cybersecurity principle:

> Never modify system configurations before understanding the root cause.

---

## 6. Server Restriction Discovery

Attempted creating:

~/.ssh

inside the Bandit account.

Observed:

Permission Denied

After investigation I realised:

Bandit intentionally prevents users from modifying their home directory.

Instead of trying to bypass restrictions, I adapted my workflow.

---

## 7. Smart Alternative Workflow

Since the remote environment prevented creating SSH configuration files, I copied the provided private key onto my own Ubuntu environment.

Using:

- nano
- chmod
- OpenSSH

I created my own identity file locally.

This demonstrated an important penetration testing mindset:

When one approach becomes impossible due to environmental restrictions, adapt your workflow without compromising security principles.

---

## 8. Successfully Authenticated

Using the copied private key, I successfully authenticated into:

bandit14

using SSH public key authentication.

---

## 9. Retrieved Current User Password

Verified successful login by accessing:

/etc/bandit_pass/bandit14

This confirmed complete success of Bandit Level 13.

# Biggest Mistake

Initially I assumed the inability to create:

~/.ssh

meant SSH authentication itself was failing.

After careful debugging, I realised the restriction was intentionally imposed by the challenge environment.

The authentication process itself was functioning correctly.

# Biggest Learning

Professional cybersecurity work rarely succeeds on the first attempt.

Systematic debugging, reading logs carefully, and adapting to environmental limitations are far more valuable than immediately searching for solutions.

Today's challenge reinforced the mindset of solving problems through investigation rather than memorisation.
