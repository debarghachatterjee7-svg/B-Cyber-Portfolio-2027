# Theory Learned

Today I learned that hexadecimal is simply another way of representing binary data. Every byte is represented by two hexadecimal digits, making binary files much easier for humans to inspect.

The `xxd` command can convert binary files into hexadecimal format and can also reverse the process using the `-r` option. This allowed me to reconstruct the original binary file from the hexadecimal dump provided in Bandit Level 12.

Another important concept was identifying file types using the `file` command. Instead of relying on filename extensions, Linux reads the file headers (magic numbers) to determine the actual file format. This is why files without extensions can still be correctly identified.

I also learned that compressed archives may contain multiple nested layers. Rather than guessing the next command, the correct workflow is to identify the current file type first, then apply the appropriate extraction utility.

# Problems Faced

Initially I was confused about how different archive formats were connected. Since several layers were nested together, I had to repeatedly identify each file before selecting the correct extraction command.

Another challenge was understanding when files needed to be renamed before decompression so that Linux utilities could recognize the correct format.

---

# How They Were Solved

By following a fixed workflow:

1. Identify the file using `file`
2. Rename if necessary
3. Use the appropriate extraction utility
4. Repeat until plain text is reached

Following this approach made the extraction process much more systematic and easier to understand.

Today's biggest lesson was that Linux tools inspect the actual contents of files instead of trusting filename extensions. Using the `file` command before performing any extraction greatly reduces mistakes and forms a reliable workflow for handling unknown files.
