# 📚 Theory Learned

Today's learning focused on how Linux provides different utilities for viewing and managing files depending on the situation.

I learned that Linux follows the philosophy that almost everything is treated as a file. Instead of relying only on the `cat` command, Linux offers utilities like `head`, `tail`, and `less`, each designed for a specific purpose.

I also learned how to use the Nano text editor to create and modify files directly from the terminal without needing a graphical interface.

Another important concept was understanding hidden files, which begin with a dot (`.`) and usually store configuration settings for users or applications.

Finally, I learned that the `file` command identifies the actual type of a file instead of trusting its extension, which is especially useful in cybersecurity when files are intentionally disguised.

---

# 💻 Commands Learned

| Command | Purpose |
|---------|---------|
| `nano` | Create and edit text files inside the terminal |
| `head` | Display the first 10 lines of a file |
| `tail` | Display the last 10 lines of a file |
| `less` | Read large files interactively |
| `file` | Identify the real type of a file |
| `man` | Open the manual page of a command |

# 🧪 Experiments & Observations

### Experiment 1

**Command**

```bash
head linux.txt
tail linux.txt
```

**Observation**

Both commands displayed exactly the same output.

**Reason**

My file contained only three lines, while both commands display ten lines by default.

---

### Experiment 2

**Command**

```bash
head logs.txt
tail logs.txt
```

**Observation**

After creating a file containing more than ten lines, `head` displayed the beginning of the file while `tail` displayed the ending.

**Conclusion**

The difference between these commands becomes visible only for larger files.

---

### Experiment 3

**Command**

```bash
less linux.txt
```

**Observation**

The file opened in interactive viewing mode.

I learned that pressing **q** exits the viewer and returns to the terminal.

---

### Experiment 4

While solving Bandit, I tried several incorrect ways of reading a filename that started with a hyphen (`-`).

The shell interpreted the filename as command options instead of an actual file, producing multiple errors.

After understanding how Linux parses command-line arguments, I successfully accessed the file using the correct relative path.

# 🌟 Biggest Learning

Today's biggest takeaway was that Linux provides different tools for different situations.

Although commands like `cat`, `head`, `tail`, and `less` all display file contents, each one is designed for a different purpose. Learning when to use each command is just as important as learning the command itself.

I also realized that carefully reading terminal error messages often provides valuable hints instead of immediately indicating failure.

---

# ❓ Biggest Confusion

The biggest challenge today occurred while solving OverTheWire Bandit Level 3. Initially, I was confused about how Linux interprets filenames that begin with a hyphen (`-`) and contain spaces. Commands like `cat` and `file` treated the filename as command-line options instead of an actual file, resulting in multiple errors.

I experimented with several approaches before understanding that Linux requires special handling for such filenames. Eventually, I learned to use the relative path (`./`) to explicitly tell the shell that I wanted to access a file in the current directory.

This experience taught me the importance of understanding how the shell interprets command-line arguments instead of assuming every typed word is treated as a filename.
