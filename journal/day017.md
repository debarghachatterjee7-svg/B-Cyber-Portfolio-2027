# 🧠 Concept 1 – What is SSH?

SSH (Secure Shell) is a secure communication protocol that allows users to remotely access another computer over an encrypted connection.

Unlike Telnet, which transmits information in plain text, SSH encrypts all communication, ensuring confidentiality and integrity.

SSH is used daily by:

- Linux System Administrators
- Cloud Engineers
- DevOps Engineers
- Ethical Hackers
- Penetration Testers
- Cybersecurity Researchers

Understanding SSH is essential because almost every Linux server is administered through it.

---

# 🔑 Concept 2 – Public Key & Private Key

Today I understood that SSH authentication is based on asymmetric cryptography.

Instead of one password, two mathematically related keys are created:

## Public Key

- Can be shared.
- Stored on remote servers.
- Used only for verification.
- Safe to copy.

## Private Key

- Must remain secret.
- Stored only on my computer.
- Used to prove my identity.
- Never transmitted over the network.

This was one of the most important concepts I learned today.

---

# 🔐 Concept 3 – SSH Authentication Flow

I learned how SSH authentication actually works.

The sequence is:

1. Client requests connection.
2. Server checks for a matching public key.
3. Server sends a cryptographic challenge.
4. Client signs the challenge using its private key.
5. Server verifies the signature using the stored public key.
6. Access is granted.

The private key is never sent to the server.

This makes SSH authentication significantly more secure than password authentication.

---

# 📂 Concept 4 – authorized_keys

The `authorized_keys` file is stored **on the server**.

Its purpose is to maintain a list of trusted public keys.

If my public key exists inside this file, the server allows me to authenticate.

I like thinking of it as a digital guest list.

---

# 🌍 Concept 5 – known_hosts

Unlike `authorized_keys`, the `known_hosts` file is stored **on the client**.

Instead of trusting users, it trusts servers.

Whenever I connect to a server for the first time, SSH records the server's identity inside this file.

Future connections compare the stored fingerprint with the server's current fingerprint.

If they don't match, SSH warns the user because the server may have changed or someone may be attempting a Man-in-the-Middle attack.

---

# 🆔 Concept 6 – SSH Fingerprints

SSH keys are extremely long.

Comparing two complete keys manually would be impractical.

Therefore SSH creates a short identifier called a fingerprint.

The fingerprint uniquely represents the original key.

Small changes in the original key produce an entirely different fingerprint.

This makes verification much easier.

# 🔒 Concept 7 – SHA-256

Today's discussion introduced SHA-256.

Although I will study hashing in much greater depth later during the Cryptography module, I now understand its purpose within SSH.

SHA-256 converts a long key into a short fingerprint.

Important properties:

- Deterministic
- One-way
- Collision resistant
- Fast verification

---

# 🎨 Concept 8 – RandomArt

One of today's most interesting concepts was RandomArt.

RandomArt converts an SSH fingerprint into a unique ASCII pattern.

Its purpose is not additional security.

Instead, it allows humans to visually compare fingerprints more easily.

This was an excellent example of improving usability without changing cryptographic security.

---

# 📄 Concept 9 – Linux File Identification

Linux does not rely on file extensions.

Instead, it examines the internal structure of a file.

Using the `file` command, Linux can determine whether something is:

- OpenSSH private key
- Public key
- Text
- Executable
- Archive
- Image

This is extremely valuable in cybersecurity because attackers frequently disguise malicious files using misleading extensions.
