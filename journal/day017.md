# What I Learned About SSH

SSH stands for Secure Shell.

SSH is a protocol that allows a user to securely connect to another computer over a network.

Instead of sending information openly, SSH encrypts communication between systems.

SSH is widely used by:

- System Administrators
- Security Engineers
- Cloud Engineers
- DevOps Engineers
- Ethical Hackers
- Cybersecurity Researchers

SSH is one of the most important technologies in Linux environments.

---

# Public Key vs Private Key

I learned that SSH uses a pair of keys.

The two keys are:

1. Public Key
2. Private Key

The Public Key can be shared.

The Private Key must never be shared.

The Private Key remains on the user's system.

The Public Key can be stored on remote systems.

The entire authentication process depends on proving ownership of the private key.

---

# Understanding Key Authentication

A server does not need my private key.

The server only needs my public key.

When I connect:

1. Server creates a challenge.
2. Challenge is sent to client.
3. Client signs challenge using private key.
4. Server verifies using public key.
5. Access is granted.

This process proves identity without transmitting the private key.

---

# Why Key Authentication Is Better Than Passwords

Passwords can:

- Be guessed
- Be leaked
- Be reused
- Be brute-forced

SSH key authentication is stronger because:

- Private keys remain local
- Authentication is cryptographic
- Strong mathematical algorithms are used
- Password reuse issues are eliminated

# authorized_keys

I learned that authorized_keys is stored on the server.

Its purpose is to store trusted public keys.

The server checks this file to determine which users are allowed to authenticate using SSH keys.

A useful way to think about it:

authorized_keys is a guest list.

If my public key is on the list:

Access is allowed.

If my public key is not present:

Access is denied.

---

# known_hosts

I learned that known_hosts is stored on the client.

Its purpose is different from authorized_keys.

authorized_keys trusts users.

known_hosts trusts servers.

When connecting to a server for the first time, SSH stores the server fingerprint in known_hosts.

Future connections compare fingerprints to ensure that the server has not changed.

---

# Understanding SSH Fingerprints

SSH keys are very large.

Comparing entire keys manually is difficult.

Therefore SSH generates a fingerprint.

A fingerprint is a short identifier derived from a key.

SSH commonly uses SHA-256 fingerprints.

A fingerprint acts similarly to a human fingerprint.

Small changes in the original key result in a completely different fingerprint.

---

# SHA-256

I learned that SHA-256 is a cryptographic hashing algorithm.

Its properties include:

- Fast computation
- One-way operation
- Strong collision resistance
- Unique output generation

SSH uses SHA-256 to create fingerprints that are easy to compare.

---

# RandomArt

RandomArt is a visual representation of an SSH fingerprint.

Instead of comparing long strings of text, users can visually compare patterns.

RandomArt does not improve cryptographic security.

Its main purpose is human readability.

I found this concept interesting because it combines security with visual verification.

# Understanding the file Command

I learned that Linux does not rely entirely on file extensions.

Instead, Linux examines file contents.

The file command helps determine the true type of a file.

Examples include:

- Text Files
- Images
- Executables
- Archives
- OpenSSH Keys

This is extremely useful in cybersecurity because attackers often disguise malicious files by changing extensions.

---

# Why File Extensions Are Not Enough

A file named:

photo.jpg

may not actually be an image.

It could be:

- Malware
- A script
- An executable

The file command examines actual file signatures rather than names.

This makes Linux much more reliable for file identification.
